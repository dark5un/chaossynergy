# ADR-006: Browser Selection and MCP Integration

**Status:** Accepted
**Date:** 2026-07-11
**Author:** Panagiotis Xynos (Master) / OrsonRius (Padawan)

## Context

The Chaossynergy desktop experience is centered around two windows: a browser and an agent terminal. The browser and agent need to be connected via MCP (Model Context Protocol) so Hermes can control the browser — navigate pages, fill forms, take screenshots, read console logs.

Additionally, secrets management needs to work seamlessly between the browser and the agent. The browser needs access to passwords and API keys; Hermes needs the same keys for its providers.

## Decision 1: Chromium in the Distrobox

**Decision:** Run Chromium inside the `agent` distrobox container, exported to the host via `distrobox-export --app chromium`.

**Alternatives considered:**
- **Host Chromium (Flatpak)** — Ships with Bluefin, but can't connect to native messaging hosts inside the container. Browser MCP extension would need to reach the MCP server inside the container across the container boundary.
- **Host Chromium (dnf)** — Pollutes the host image. Harder to update independently.
- **Container Chromium (chosen)** — Everything stays inside the container. The Chrome extension (Browser MCP) talks to the MCP server (Node.js `npx` process) on localhost inside the container. Browserpass extension talks to browserpass-native (Go binary) on the same machine.

**Rationale:**
- The browser and its MCP dependencies are co-located in one container
- No cross-container networking needed — WebSocket between Chrome extension and MCP server stays on localhost
- Native messaging (browserpass) works natively without path trickery
- The browser is disposable with the container — reset everything with `distrobox rm agent`

## Decision 2: Browser MCP for Agent-Browser Control

**Decision:** Use [Browser MCP](https://browsermcp.io/) (Chrome extension + `npx @browsermcp/mcp`) for Hermes-to-browser automation.

**Alternatives considered:**
- **Playwright MCP** — Creates headless browser instances. Can't use the user's logged-in sessions.
- **browser-use** — Cloud-based, requires API key. Not local-first.
- **Hermes built-in browser tools** — Already exist, but use a different browser instance. Browser MCP reuses the user's actual Chromium with its sessions and cookies.

**Rationale:**
- Uses the user's real browser session — logged-in, with cookies and profiles
- Avoids bot detection — the browser is a real user's browser, not a headless agent
- All local — no cloud dependency, no API keys
- Hermes connects via stdio MCP transport: `npx @browsermcp/mcp@latest`

## Decision 3: Browserpass for Secrets in the Browser

**Decision:** Use [browserpass](https://github.com/browserpass/browserpass-extension) (Chrome extension + `browserpass-native` Go binary) to give the browser access to the `pass` password store.

**Alternatives considered:**
- **Built-in Chromium password manager** — No cross-app access (Hermes can't read Chromium passwords).
- **KeePassXC + KeePassXC-Browser** — Similar function, but uses a different store format.
- **Bitwarden** — Cloud-dependent. Chaossynergy is local-first.
- **browserpass (chosen)** — Uses the same `pass` store Hermes reads from. Simple, Unix-native, GPG-encrypted.

**Rationale:**
- A single `~/.password-store` is shared between host and container
- Hermes reads API keys: `pass show hermes/openrouter`
- Browserpass reads the same store: browserpass Chrome extension → browserpass-native → pass
- GPG encryption means secrets are encrypted at rest
- No cloud dependency, no subscription

## Consequences

- Chromium inside the container is ~330MB — the agent image is large but self-contained
- Chrome extensions are force-installed via managed policies — user can't accidentally remove them
- Browser MCP requires the user to click "Connect" in the extension once per session (or the tab stays connected)
- pass requires a one-time GPG key setup — guided by `user-setup.sh`
- Hermes config gets an `mcp_servers.browsermcp` entry generated by `user-setup.sh`
